Below is a remediation plan in English, ordered to reduce risk and avoid rewriting the same areas twice.

# Remediation Plan

## Goal

Reduce structural complexity, remove fake architectural separation, move expensive computation off the render path and off the main thread, make persistence predictable, replace fake observability, and cut bundle weight.

## Guiding principle

Do not “improve architecture” in the abstract. Each change must remove measurable cost:

* fewer cross-layer imports
* smaller main bundle
* fewer main-thread blocking tasks
* fewer pointless writes
* fewer silent failures
* simpler module ownership

---

# Phase 1 — Collapse the fake layers into a simpler structure

## Target structure

Use three real boundaries only:

* **Geometry domain**: geometry rules, solar/shading math inputs and outputs, validators, serializers
* **Application state**: store, commands, persistence coordination, async job orchestration
* **UI**: React components, view hooks, presentation-only mapping

## Tasks

1. Define ownership rules:

   * domain imports nothing from app or UI
   * state can import domain
   * UI can import state and domain DTOs only when truly needed
2. Remove cross-layer backreferences:

   * eliminate `application -> app`
   * eliminate `state -> application presentation`
   * eliminate any circular dependency hidden by folder naming
3. Replace folder-deep pseudo-abstractions with concrete modules:

   * remove decorative mappers/selectors that only rename fields
   * merge thin wrappers where they add no isolation
4. Split “god hooks” into:

   * pure domain functions
   * state orchestration
   * UI adapter hooks
5. Add import-boundary enforcement in lint rules

## Deliverables

* new module map
* dependency rules documented in one page
* import-boundary lint checks
* reduced cross-layer imports to near zero

## Acceptance criteria

* no import from UI/app into domain
* no import from presentation layer back into state/app core
* large hooks reduced into smaller modules with single responsibility
* developers can locate logic by ownership without guessing

## Risk

This can become a cosmetic folder shuffle. It only counts if imports and responsibilities actually change.

---

# Phase 2 — Move shading and annual simulation off render and off the main thread

## Goal

No heavy geometry or solar simulation inside React render lifecycle.

## Tasks

1. Identify all expensive compute paths:

   * roof shading grid
   * annual simulation
   * heatmap generation
   * any cache-key serialization that walks full geometry
2. Extract compute into pure worker-safe functions:

   * deterministic inputs
   * no React/state/store access
   * no DOM/browser coupling
3. Introduce Web Worker execution:

   * one worker for shading/simulation pipeline
   * message-based job API
   * cancellation support for stale jobs
4. Replace render-time `useMemo` compute with async job orchestration:

   * UI submits job
   * state stores job status/result/error
   * UI renders loading/progress/result
5. Replace geometry-string cache keys with revision-based keys:

   * project revision
   * roof revision
   * obstacle revision
   * settings revision
6. Add result caching at worker boundary, not inside render hooks
7. Add profiling around:

   * job duration
   * worker queue time
   * main-thread blocking reduction

## Deliverables

* worker module
* typed job protocol
* cancellation/versioning logic
* performance benchmark before/after

## Acceptance criteria

* no heavy compute runs directly in render hooks
* no large geometry serialization just to form cache keys
* UI remains responsive during shading/simulation
* stale results cannot overwrite newer state

## Risk

The common failure here is moving code into a worker but keeping half the orchestration tangled in React hooks. Keep React dumb.

---

# Phase 3 — Redesign persistence to debounce / idle / commit-based writes

## Goal

Persistence should be intentional, sparse, and failure-aware.

## Tasks

1. Define what actually needs persistence:

   * persistent project data
   * ephemeral UI selection state should not be written unless explicitly required
2. Introduce persistence policy:

   * debounce for frequent edits
   * idle callback when safe
   * explicit commit on major actions (save/import/export)
3. Stop writing the whole project on every state change
4. Persist only normalized serializable state
5. Add storage error handling:

   * quota exceeded
   * unavailable storage
   * blocked access / private mode edge cases
6. Add corruption handling:

   * versioned schema
   * migration path
   * invalid payload fallback
7. Surface persistence status in state:

   * clean / dirty / saving / failed

## Deliverables

* persistence adapter
* debounce/idle scheduler
* schema versioning and migration
* storage error reporting path

## Acceptance criteria

* selection-only changes do not trigger pointless project writes
* write frequency drops sharply under active editing
* storage failures are visible and non-fatal
* restored state is versioned and validated

## Risk

Do not bury persistence logic inside store subscriptions with hidden side effects. Make it explicit and testable.

---

# Phase 4 — Replace fake observability or remove it

## Goal

Either introduce real telemetry/error reporting, or delete the pretend layer.

## Options

### Option A — Minimal real observability

Use a small, explicit telemetry interface:

* `captureError`
* `captureWarning`
* `captureMetric`
* `captureBreadcrumb`

Backends:

* console in dev
* remote provider in production, if the product actually needs it

### Option B — Remove observability abstraction entirely

If there is no backend and no operational use, remove the layer and use local logging only.

## Tasks

1. Audit current event-based observability calls
2. Classify each use:

   * user-facing recoverable error
   * developer diagnostic
   * operational signal
3. Remove `CustomEvent`-based fake telemetry
4. Wire error boundaries to real reporting or useful fallback UI
5. Ensure all async worker/storage/network failures flow through one path

## Deliverables

* single telemetry interface or full removal
* meaningful error boundary fallback
* production/dev behavior clearly separated

## Acceptance criteria

* no “observability” that only dispatches browser events into nowhere
* runtime crashes do not produce blank screens
* critical errors have a single reporting path

## Risk

Fake observability is worse than none because it creates false confidence.

---

# Phase 5 — Cut bundle size with lazy loading and feature boundaries

## Goal

The initial load should contain only what the default user path needs.

## Target candidates for lazy loading

* 3D
* heatmaps
* charts
* debug tools
* heavy map extras, if not needed at first paint

## Tasks

1. Measure bundle composition
2. Split by feature entry points:

   * base app shell
   * map core
   * 3D tools
   * heatmap/simulation visualization
   * charts/reporting
   * debug/developer tools
3. Use route-level and interaction-level lazy loading
4. Delay heavy library initialization until feature entry
5. Remove dead code and unused utility layers
6. Verify no shared chunk accidentally re-pulls everything back into the main bundle

## Deliverables

* chunking strategy
* lazy-loaded feature boundaries
* updated build report
* regression guard in CI for bundle size

## Acceptance criteria

* significant reduction in main entry chunk
* optional features do not load before first use
* no visible regression in feature behavior after lazy loading

## Risk

Superficial code splitting often fails because one shared utility imports heavy dependencies and drags them back into the main chunk.

---



# Definition of done

The work is done when:

* the architecture is simpler in practice, not just renamed
* the UI no longer blocks on shading/simulation
* persistence is sparse and failure-aware
* observability is real or deleted
* heavy features are not in the initial bundle
