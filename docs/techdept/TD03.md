# Technical Debt / Architecture Erosion Note

## Title
Architecture Erosion Around `useSunCastController.ts` and `projectState.reducer.ts`

## Status
Open

## Severity
High

## Summary

The codebase shows architectural intent, but the runtime boundaries are still weak.

Two files currently carry too much responsibility:

- `src/app/hooks/useSunCastController.ts` (~715 lines)
- `src/state/project-store/projectState.reducer.ts` (~541 lines)

These files act as structural pressure points.  
They are not just “large files”. They are evidence that the current boundaries are too soft and that multiple concerns are still collapsing into central orchestration and central state mutation.

This is architecture erosion, not just code style debt.

---

## Problem Statement

The codebase declares boundaries such as:

- geometry domain
- app state
- app orchestration
- map interaction
- rendering
- tutorial
- sun tools / simulation

But those boundaries are only partially enforced.

In practice:

- `useSunCastController.ts` behaves like the hidden application layer
- `projectState.reducer.ts` behaves like a mixed document + session + UI reducer

This means the folder structure suggests modularity, while the runtime still depends on a few oversized coordination points.

---

## File 1: `useSunCastController.ts`

### Current role
This hook appears to coordinate:

- project state access
- selection/editing flow
- view-model composition
- shading flow
- annual simulation flow
- tutorial actions
- keyboard/UI actions
- rendering-related orchestration

### Why this is debt
A single hook at this size becomes the real system boundary, regardless of folder structure.

This creates the following failure modes:

- feature creep accumulates in one place
- merge conflicts concentrate in one file
- UI flow, domain decisions, and orchestration become entangled
- changes are harder to reason about and test
- architectural boundaries become advisory instead of enforced

### Diagnosis
`useSunCastController.ts` is effectively a god hook.

It is not a thin composition layer anymore.  
It is acting as:

- coordinator,
- application service,
- feature model,
- state adapter,
- and workflow owner.

That is too much.

---

## File 2: `projectState.reducer.ts`

### Current role
This reducer currently appears to manage a mixed set of concerns, including:

- persisted project data
- active ids
- selection state
- drawing drafts
- editing flags
- interaction/session state

### Why this is debt
A reducer should have a coherent ownership boundary.

Right now the reducer mixes:

1. **document state**  
   data that should be saved/shared/loaded

2. **editor session state**  
   temporary interaction state that should not be persisted

That creates hidden coupling.

### Consequences
- persisted and transient concerns evolve together
- editor behavior can destabilize save/share logic
- reducer complexity grows with every feature
- migrations and state sanitization become harder
- testing becomes broader and less precise

### Diagnosis
`projectState.reducer.ts` is not just “too long”.  
It has a broken ownership model.

---

## What Is Missing

### 1. Hard separation between persisted document and editor session
This is the biggest missing boundary.

There should be a clear split between:

#### Project Document
Persisted, canonical, shareable data:
- footprints
- constraints
- obstacles
- sun/shading settings

#### Editor Session
Ephemeral runtime data:
- selected ids
- active item
- draw draft
- selected vertex/edge
- drag state
- interaction flags
- map/orbit temporary state

That separation is currently incomplete.

---

### 2. A real application layer
At the moment the orchestration is too concentrated in one hook.

What is missing is an explicit application layer with smaller, focused models such as:

- `useSidebarModel()`
- `useCanvasModel()`
- `useTutorialModel()`
- `useAnalysis()`
- `useEditorSession()`

Without this split, the controller remains the hidden runtime architecture.

---

### 3. Boundary-level contracts
The current structure needs stronger contracts between:

- document state
- editor session
- analysis pipeline
- rendering adapters
- UI presentation models

These contracts should be narrow and typed.

Right now the boundaries are present conceptually, but they are not strict enough to prevent leakage.

---

### 4. Derived state isolation
Solved roofs, shading results, heatmaps, annual simulation output, warnings, and analysis diagnostics should live in a dedicated derived/analysis boundary.

They should not drift into persisted state or central controller glue.

This separation is still too soft.

---

### 5. Thin adapters
Rendering and map integration should remain adapter layers.

They should not become secondary homes for product logic, fallback behavior, or hidden transformation rules.

That line is currently at risk.

---

## Why This Matters

This debt does not only affect readability.

It materially affects:

- delivery speed
- regression risk
- feature conflict rate
- testability
- onboarding cost
- confidence in refactoring

The codebase may look modular from the outside, but changes still converge toward the same central files.

That is a scaling problem.

---

## Target Architecture

### 1. Project Document
Owns persisted canonical state only.

Suggested boundary:
`src/domain/project-document/*`

Contains:
- types
- reducer
- schema
- migrations
- storage
- share/import/export

Does **not** contain:
- selection
- drafts
- drag state
- temporary map state
- analysis output

---

### 2. Editor Session
Owns temporary runtime editing state only.

Suggested boundary:
`src/application/editor-session/*`

Contains:
- active selection
- selected vertex/edge
- drafts
- drag state
- interaction flags
- temporary map/editor state

Does **not** own persisted project data.

---

### 3. Analysis Pipeline
Owns derived computed state.

Suggested boundary:
`src/application/analysis/*`

Contains:
- solved roofs
- shading inputs
- shading results
- annual simulation results
- diagnostics/warnings

Does **not** own persistence or UI behavior.

---

### 4. Presentation Models
Own thin screen-facing composition.

Suggested boundary:
`src/application/presentation/*`

Examples:
- `useSidebarModel()`
- `useCanvasModel()`
- `useTutorialModel()`

These should shape UI inputs and outputs, not become workflow engines.

---

### 5. Adapters
Own framework/platform-specific integration.

Suggested boundaries:
- `src/adapters/map/*`
- `src/adapters/rendering/*`

These should stay thin.

---

## Fix Plan

## Phase 1 — Split state ownership
### Goal
Separate persisted project data from editor session state.

### Actions
- create `project-document` boundary
- create `editor-session` boundary
- move selected ids, drafts, drag/interactions out of `projectState.reducer.ts`
- keep `projectState.reducer.ts` only for persisted document concerns, or replace it with a document reducer

### Outcome
A real distinction between what is saved and what is only runtime state.

### Effort
2–4 weeks

---

## Phase 2 — Decompose `useSunCastController.ts`
### Goal
Remove the hidden god hook.

### Actions
Split into focused models:

- `useProjectDocument()`
- `useEditorSession()`
- `useAnalysis()`
- `useSidebarModel()`
- `useCanvasModel()`
- `useTutorialModel()`

Move feature-specific orchestration into the correct layer.

### Outcome
Smaller composition points, clearer ownership, easier testing.

### Effort
2–3 weeks

---

## Phase 3 — Move derived data into analysis boundary
### Goal
Prevent derived computation from leaking into store and controller glue.

### Actions
- centralize solved roof computation
- centralize shading orchestration
- centralize annual simulation orchestration
- define typed outputs for analysis results and warnings

### Outcome
A cleaner split between canonical input and computed output.

### Effort
1–2 weeks

---

## Phase 4 — Tighten adapter boundaries
### Goal
Keep rendering and map code thin.

### Actions
- remove product logic from map/rendering adapters
- keep only lifecycle, bridge, sync, and event emission there
- define stable typed contracts for adapter inputs

### Outcome
Lower rendering and interaction coupling.

### Effort
2–3 weeks

---

## Phase 5 — Add boundary-focused tests
### Goal
Protect the new structure.

### Actions
Add tests around:
- load/save/share
- selection/edit flow
- shading run
- annual simulation run
- reset/recovery flows

Test boundaries, not just helpers.

### Outcome
Refactors become safer.

### Effort
2–4 weeks

---

## Concrete Refactoring Direction

## Replace this:
```text
SunCastScreen
  -> useSunCastController()
  -> everything
````

## With this:

```text
SunCastScreen
  -> useProjectDocument()
  -> useEditorSession()
  -> useAnalysis(document, session)
  -> useSidebarModel(document, session, analysis)
  -> useCanvasModel(document, session, analysis)
  -> useTutorialModel(document, session)
```

This is the real structural cut.

---

## Risks If Left Unfixed

If this remains as-is, likely outcomes are:

* more logic added to `useSunCastController.ts`
* more transient state added to `projectState.reducer.ts`
* higher feature collision rate
* more defensive glue code
* harder testing
* slower onboarding
* “modular” architecture that becomes increasingly fake over time

---

## Decision

This debt should be treated as a structural remediation item, not as optional cleanup.

The problem is not that the files are large.
The problem is that they currently hold responsibilities that should belong to different architectural boundaries.

---
