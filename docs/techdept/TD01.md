## Verdict

The main structural failure is this:

* `useSunCastController` is acting as the system brain, but it mixes document state, editor session, analysis, rendering, and UI flow.
* `useProjectStore` pretends to be a domain store, but actually mixes persisted project data with temporary editor state.

That needs to be cut into **5 parts**.

---

## 1. Project Document — the only source of persisted truth

This should contain only what the user actually edits and what you save/share.

**Keep here:**

* footprints
* constraints
* obstacles
* sun projection settings
* shading settings

**Do not keep here:**

* selected ids
* drawing draft
* camera/orbit state
* solved roofs
* heatmaps
* annual simulation output
* worker state
* transient UI flags

**Move into this boundary:**

* most of `state/project-store/projectState.types.ts`
* persisted parts of `projectState.reducer.ts`
* `projectState.schema.ts`
* `projectState.storage.ts`
* `projectState.share.ts`
* `projectState.sanitize.ts` → but rewritten properly

**Suggested location:**

```text
src/domain/project-document/*
```

---

## 2. Editor Session — temporary runtime editing state

This is UI/editor session state. It is not part of the saved project.

**Keep here:**

* active footprint / obstacle
* selected footprint ids / obstacle ids
* draw draft / obstacle draw draft
* selected vertex / selected edge
* drag state
* interaction errors
* orbit enabled
* map initialized / map pitch / map bearing

**Move into this boundary:**

* `useSelectionState.ts`
* editor-specific parts of `useConstraintEditor.ts`
* transient state currently hidden inside `useSunCastController.ts`
* non-persisted parts of `projectState.reducer.ts`

**Suggested location:**

```text
src/app/editor-session/*
```

---

## 3. Analysis Pipeline — derived, computed state only

This is where all computed outputs belong.

**Keep here:**

* solved roofs
* selected roof inputs
* shading scene
* live shading result
* annual simulation result
* diagnostics / warnings

**Move into this boundary:**

* `useSolvedRoofEntries.ts`
* `useRoofShading.ts`
* `useAnnualRoofSimulation.ts`
* `useSelectedRoofInputs.ts`

**Suggested location:**

```text
src/app/analysis/*
```

Important rule:
This layer should return **typed results**, not `null`, silent fallback, or “best effort”.

---

## 4. Map and Rendering Adapters — platform-specific only

MapLibre and Three.js should not contain product logic.

### Map adapter should do only:

* click / hover / drag wiring
* hit testing
* camera state sync
* source/layer sync
* emit editor intents

### Rendering adapter should do only:

* mesh layer lifecycle
* heatmap overlay
* worker bridge
* renderer lifecycle

**Move into this boundary:**

* `app/features/map-editor/MapView/*` → adapter layer
* `rendering/roof-layer/*` → rendering adapter layer

**Suggested location:**

```text
src/adapters/map/*
src/adapters/rendering/*
```

And this matters:
`MapView` should stop taking a huge bag of props. It should receive **one canvas model**.

---

## 5. Presentation / Feature Models — thin UI composition only

These are screen-facing compositional models for sidebar, canvas, tutorial, etc.

**Keep here:**

* screen-level composition
* UI-facing actions
* view model shaping

**Do not keep here:**

* persistence logic
* geometry solving
* storage/share logic
* rendering implementation

**Replace the god hook with:**

* `useSidebarModel()`
* `useCanvasModel()`
* `useTutorialModel()`

Instead of one giant controller.

---

# What the flow should look like

## Current bad shape

```text
SunCastScreen
  -> useSunCastController()
  -> everything
```

## Better shape

```text
SunCastScreen
  -> useProjectDocument()
  -> useEditorSession()
  -> useAnalysis(document, session)
  -> useSidebarModel(document, session, analysis)
  -> useCanvasModel(document, session, analysis)
  -> useTutorialModel(document, session)
```

That is the real architectural cut.

---

# The most important cuts

## Cut 1 — split persisted document from transient session

Today the store mixes both. That is wrong.

**Target:**

```text
domain/project-document/
  projectDocument.types.ts
  projectDocument.reducer.ts
  projectDocument.storage.ts
  projectDocument.share.ts
  projectDocument.schema.ts
  projectDocument.migrations.ts

app/editor-session/
  editorSession.types.ts
  editorSession.reducer.ts
  editorSession.commands.ts
  editorSelection.ts
  editorDrawing.ts
```

**Rule:**

* `project-document` = persisted
* `editor-session` = ephemeral

This is the single most important cut.

---

## Cut 2 — kill `useSunCastController`

That file is artificial centralization. It is not architecture.

Split it by output model, not by random helpers:

```text
app/presentation/
  useSidebarModel.ts
  useCanvasModel.ts
  useTutorialModel.ts
```

Each one should build only what one UI area needs.

---

## Cut 3 — make analysis a real pipeline

Right now solve/shading/simulation logic is scattered.

Create one proper analysis boundary:

```text
app/analysis/
  analysis.types.ts
  deriveSolvedRoofs.ts
  deriveShadingScene.ts
  useLiveShading.ts
  useAnnualSimulation.ts
  deriveSelectedRoofInputs.ts
  deriveHeatmapMode.ts
```

**Input:** `ProjectDocument`, `EditorSession`
**Output:** `AnalysisState`

---

## Cut 4 — introduce a `CanvasModel`

`MapView` currently has too much surface area.

Instead of passing a giant prop bag, define one model:

```ts
type CanvasModel = {
  documentView: ...
  selectionView: ...
  drawingView: ...
  renderScene: ...
  cameraState: ...
  handlers: ...
}
```

Then `MapView` receives one coherent model instead of dozens of unrelated props.

# Dependency rules

Without hard dependency rules, they will rebuild the same mess.

## Allowed

* `domain` → nothing
* `application` → `domain`
* `adapters` → `application`, `domain`
* `features/screens` → `application`, `adapters`, `domain`

## Forbidden

* `domain` → React / MapLibre / storage / browser APIs
* `application` → Three.js / MapLibre / DOM
* `adapters/rendering` → business rules
* `features` → direct storage/share/schema mutation

---

# What not to do

Do **not**:

* create another giant controller hook
* create more “helpers” without changing boundaries
* keep sanitizing broken state instead of validating contracts
* keep hiding failures with fallback chains

Do:

* one persisted document
* one editor session
* one analysis pipeline
* one canvas model
* one sidebar model

---

# Implementation order

## Phase 1

1. Extract `project-document`
2. Extract `editor-session`

## Phase 2

4. Break `useSunCastController` into `useSidebarModel`, `useCanvasModel`, `useTutorialModel`

6. Reduce `MapView` to a single `CanvasModel`
7. Reduce renderer input to a single `RenderScene`
